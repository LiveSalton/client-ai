# Wilder 清理重构

分析阶段可以激进，删除阶段必须保守。

## 适用场景

- 死代码清理
- 删除未使用导出或依赖
- 收敛重复工具函数或组件
- 功能稳定后的仓库整理工作

## 不适用场景

- 正在进行功能开发且范围边界不清晰
- 临近发布且没有恢复窗口
- 项目缺少有效验证手段
- 无法判断代码是否属于公共接口

## 检测命令

运行适合当前技术栈的检测工具。

```bash
npx knip
npx depcheck
npx ts-prune
npx eslint . --report-unused-disable-directives
vulture src/
deadcode ./...
cargo +nightly udeps
```

如果没有专用工具，就手动搜索引用，并把动态导入、字符串查找、配置注册和外部调用方都纳入检查范围。

## 风险分级

- `SAFE`：内部工具、私有辅助函数、明显未使用的依赖
- `CAUTION`：组件、路由、中间件、被间接引用的文件
- `DANGER`：公共接口、入口文件、配置文件、共享契约

## 清理循环

1. 先用测试或构建建立绿色基线。
2. 先删除最安全类别中的一小批项。
3. 立即重新验证。
4. 如果验证失败，回退该批改动并下调可信度。
5. 当前批次被证明安全后再继续。

## 重复代码收敛

- 优先保留命名更清晰、测试更完善、依赖更少的实现。
- 先更新引用，再删除旧实现。
- 清理动作与行为变更应分开处理。

## 安全检查单

- 工具或手动搜索确认代码未被使用
- 已检查动态引用
- 已确认公共接口属性
- 删除后验证通过
- 条件允许时，每批改动都有版本控制检查点

## 总结模板

```markdown
死代码清理
-----------------
删除: 12 个未使用函数
删除: 3 个未使用文件
删除: 5 个未使用依赖
跳过: 2 个高风险项
结果: 测试和构建仍然通过
```

## 规则

- 不能只凭工具输出直接删除。
- 一次只处理一个类别：依赖、导出、文件、重复代码。
- 没把握就跳过，不要强行自信。
- 不要把大范围重构混入清理流程。
