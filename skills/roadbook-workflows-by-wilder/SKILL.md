# Wilder Roadbook 工作流

编写 roadbook（路程指南）：编码前作为落地前终稿，用人类友好的方式讲清背景、目标、路线、影响和注意事项；代码落实后作为知识库使用指南，补齐如何使用、如何验证、常见问题和后续衔接。`roadbook` 给人查阅理解，`OpenSpec` 给程序员和 AI 精确执行。

## 适用场景

- 涉及多个文件或子系统的新功能
- 需要明确顺序、回滚策略或分阶段推进的重构
- 适合分阶段交付的架构调整
- 依赖、风险或测试策略尚不清晰的任务
- 需要为后续 `OpenSpec` 提案提供人类可读背景和决策依据的任务
- 代码落实后需要沉淀使用方式、验证结果和注意事项的任务
- 需要作为知识库长期给他人查阅的任务

## 工作流程

1. 分析请求并定义预期结果。
2. 列出约束、假设和成功标准。
3. 检查当前业务背景、用户路径、相似经验和必要技术约束。
4. 将任务拆解为读者能理解的路线，优先解释“为什么这样走”和“会影响什么”。
5. 标出风险、边界情况、验证方式和注意事项。
6. 为每个阶段写清目标、判断依据、验收点和可逆处理方式。
7. 将 roadbook 保存到 `doc/roadbook/`，作为后续 `OpenSpec` 提案输入。
8. 代码落实后回写同一份 roadbook，补齐使用指南、验证结果、注意事项和后续衔接。

## Roadbook 格式

```markdown
# Roadbook: <topic>

## 概述
- 目标摘要

## 需求
- 业务需求
- 约束与假设

## 背景与影响
- 为什么要做
- 会影响哪些用户、场景或流程
- 不做会有什么问题

## 执行路线
### 阶段 1: <name>
1. 步骤名称
   - 目标: 这个阶段要解决什么
   - 做法: 用非技术语言说明要怎么推进
   - 原因: 为什么这样做
   - 依赖: 前置步骤或无
   - 影响范围: 影响哪些页面、流程、用户或角色
   - 验收点: 人类读者如何判断完成
   - 可逆处理: 出问题时如何停下、回退或绕开
   - 风险: low / medium / high

## 验证方式
- 需要人工确认什么
- 需要工具或测试确认什么
- 哪些验证未做以及原因

## 使用指南
- 如何启用、调用或访问
- 关键入口或操作步骤
- 常见注意事项

## 验证结果
- 已运行的验证
- 未运行的验证及原因
- 已知风险或后续衔接

## 风险与缓解
- 风险及对应措施

## OpenSpec 输入映射
- proposal.md: 可复用的背景、目标、非目标和影响范围
- design.md: 可复用的关键决策、必要技术约束和风险
- tasks.md: 可复用的阶段、验收点和验证方式
- spec.md: 可复用的需求、边界和验收标准

## 成功标准
- 可观察的完成标准
```

## 产物规范

- roadbook 输出必须保存为 Markdown 文件，放在 `doc/roadbook/` 目录下，建议命名为 `doc/roadbook/<topic>-roadbook.md`。
- 编码前，文档必须作为落地前终稿，包含：背景、目标、约束、假设、影响范围、执行路线、验收点、可逆处理、验证方式、风险、成功标准。
- 代码落实后，必须回写同一份 roadbook，补齐：使用指南、验证结果、注意事项、已知风险和后续衔接。
- 该文档是 `OpenSpec` 提案的输入，不是提案本身，也不能替代 `proposal.md` / `spec.md` / `tasks.md` / `design.md`。
- 如果后续要进入 `OpenSpec` 流程，应优先复用这份 roadbook 中的背景、决策依据、影响范围、阶段、验收点和验证方式。

## 质量要求

- 优先写清背景、目标、影响和判断依据，避免一上来堆技术名词。
- 技术细节只在理解必要时出现，并用普通语言解释；复杂实现细节留给 `OpenSpec`。
- 按依赖顺序规划，而不是按顺手程度规划。
- 在可行时保持阶段可独立交付。
- 显式写出边界情况，不要藏在实现细节里。
- 必须包含具体验证方案。
- 必须写清每个阶段的目标、做法、验收点和可逆处理。
- 落实后必须让后续维护者能据此理解如何使用和验证结果。

## 风险信号

- 步骤只有技术动作，没有解释业务目的、影响范围或判断依据
- 阶段过大，无法独立验证
- 缺少可逆处理或风险说明
- 没有验证方式
- 把无关重构混入当前需求
- roadbook 写得像技术设计文档，非技术读者无法理解
