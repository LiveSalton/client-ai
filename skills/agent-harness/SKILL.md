# Agent Harness 搭建

为代码项目构建 AI Agent 文档体系，使任何新 Agent 打开项目后能通过 AGENTS.md 快速掌握产品逻辑、技术架构、功能目录和接手方式。

## 核心原则

在开始任何工作前，必须内化以下原则：

1. **保留原内容** — 已有文档（AGENTS.md、README.md 等）原封不动保留，只追加新内容
2. **AGENTS.md 精简** — 必须包含 `## 项目概览` + 门禁规则 + 文档索引指针，不放详细内容
3. **详细内容由子文档承载** — 所有详细内容放在 `doc/` 目录下
4. **只写约束规则** — 文档只写"必须/禁止"类约束，不写读代码就能知道的实现细节（包路径、继承关系、API 调用方式等）
5. **不写动态记录** — 不写需要频繁更新的数据（提案数量、已知 bug、当前进度等）
6. **信息不重复** — 每条信息只在一个权威文档中出现，其他文档引用它
7. **先扫描后动手** — 必须充分理解项目后再规划文档结构

## 步骤

### 1. 扫描项目

扫描以下维度，建立项目全貌：

- 项目根目录结构（`ls -la`）
- 技术栈（构建文件、依赖声明、框架配置）
- 架构模式（目录组织、分层方式、基类/接口）
- 现有文档（README、CONTRIBUTING、doc/、wiki 等）
- 是否有 spec/proposal 管理系统（openspec/、rfc/、adr/ 等）
- 已有 AGENTS.md 或类似文件的当前内容
- 国际化、主题/设计系统、CI/CD 等横切关注点

**关键**：扫描阶段只读不写。充分理解后再规划。

### 2. 规划文档结构

根据扫描结果规划文档体系。标准模板：

```
AGENTS.md                          # 精简入口（~30-50 行）
doc/
  project-structure.md             # 项目结构约束（文件规模限制、页面路书）
  project-code-style.md            # 编码约束（架构规则、命名约定）
  project-theme.md                 # 设计/主题规范（如有设计系统）
  project-i18n-standard.md         # 国际化标准（如多语言）
  project-workflow.md              # 工作流规范（如有协作流程）
  project-opsx.md                  # Spec/提案工作流说明（如有）
```

根据项目实际情况裁剪：
- 没有国际化 → 跳过 i18n
- 没有设计系统 → 跳过 theme
- 没有 spec 管理 → 跳过 opsx
- 有特殊关注点 → 新增对应文档（如 project-security.md、project-api.md）

将文档结构方案呈现给用户，获得确认后再写入。

### 3. 编写 AGENTS.md

AGENTS.md 是入口文件，控制在 30-50 行，包含三部分：

**a. `## 项目概览`**（不超过 100 字）
必须使用二级标题 `## 项目概览`，用一段话概括当前项目定位、技术栈、架构模式或发行渠道，描述不超过 100 字。

**b. 最高门禁**（核心规则）
根据项目特点提炼约束规则，常见门禁包括：
- 禁止主动构建/编译/发布（除非用户要求）
- 禁止主动编写测试代码（除非用户要求）
- 最小修改原则，禁止顺手重构
- 写入授权机制（如用户消息包含特定关键字才可写入）
- Spec 文档同步规则
- 改动前必须阅读相关 doc/ 文档，向用户报告方案 + 改动内容 + 涉及文件

**c. 规范文档索引**（导航表）
表格列出 doc/ 下每个文件的名称、文件路径、内容说明。

### 4. 编写 doc/ 子文档

逐个编写子文档，每个文档遵循：

- **只写约束规则**（必须做 X、禁止做 Y）
- **不写实现细节**（代码中可查到的包路径、类继承、API 调用等）
- **不写动态数据**（当前数量、进度、已知问题等）
- **与其他文档不重复**（引用而非复制）

各文档典型内容：

| 文档 | 写什么 | 不写什么 |
|------|--------|----------|
| project-structure.md | 文件规模限制、页面/模块路书 | 目录树、依赖列表、技术栈表 |
| project-code-style.md | 架构约束、编码禁忌 | 常用模式、代码示例、基类签名 |
| project-theme.md | 设计语言约束、Token 命名规则 | 具体颜色值（查主题文件可知） |
| project-i18n-standard.md | 翻译流程、命名约定、覆盖要求 | 已有语言列表（查资源目录可知） |
| project-workflow.md | 协作规则、提交约定 | Git 命令教程 |
| project-opsx.md | 提案工作流、文档同步规则 | 当前提案列表 |

### 5. 更新配置上下文

如果项目使用 spec 管理系统（如 openspec），在其配置中补充项目上下文字段（技术栈、架构、设计语言等），确保 spec 工具能感知项目背景。

### 6. 验证

完成所有文档后进行交叉验证：

- 扫描所有文档，检查是否有重复内容
- 确认 AGENTS.md 包含 `## 项目概览` 章节，且该章节描述不超过 100 字
- 确认 AGENTS.md 中没有详细内容（应只有指针）
- 确认没有动态记录（需要频繁更新的数据）
- 确认所有约束规则都是"必须/禁止"类，不是实现描述
- 向用户呈现所有文件草稿，获得确认后再写入仓库

## 陷阱

- **不要重写已有文档** — 用户非常在意保留原始内容，只追加新内容
- **不要把 AGENTS.md 写胖** — 技术栈表、目录树、依赖列表等详细内容应放在子文档中
- **不要写实现细节** — "ViewModel 继承 BaseMvvmActivity<VM, VB>"这种从代码就能看出来的不要写
- **不要写动态记录** — "当前有 24 个提案"这种每次都要更新的数据不要写
- **不要让门禁规则在多处重复** — 门禁只在 AGENTS.md 中定义，子文档引用它
- **不要跳过用户确认** — 先呈现草稿，用户确认后再写入
- **文档索引标题避免与门禁规则冲突** — 如门禁说"改动前阅读相关文档"，则索引用"规范文档索引"而非"修改代码前必须阅读"
