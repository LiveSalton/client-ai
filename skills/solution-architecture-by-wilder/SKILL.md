# Wilder 方案架构

设计在持续变化下仍然易理解、易维护、可扩展的系统。

## 适用场景

- 新系统或大型功能设计
- 涉及模块边界调整的大型重构
- 涉及扩展性、可靠性或安全性权衡的决策
- 需要先明确职责、接口或数据流的任务

## 评审流程

1. 评估当前结构、约定和技术债。
2. 收集功能性与非功能性需求。
3. 定义职责、接口、数据流与集成点。
4. 比较设计选项并记录权衡。
5. 推荐能满足当前需求且不阻碍后续演进的最简方案。

## 核心原则

- 高内聚、低耦合
- 清晰边界与显式接口
- 在合适情况下优先无状态设计
- 默认安全的信任边界
- 模块应可测试、可观测、易推理

## 决策模板

```markdown
## 决策: <topic>

### 背景
- 问题与约束

### 选项
- 选项 A
- 选项 B

### 权衡
- 优点
- 缺点

### 推荐方案
- 选择及理由

### 后续影响
- 后续工作
- 运维或迁移影响
```

## 设计检查单

- 职责已分配到明确模块
- 系统边界上的数据流已写清
- 接口契约明确
- 错误处理策略明确
- 安全假设已说明
- 已识别性能与扩展约束
- 已考虑监控、发布与回滚

## 反模式

- 上帝对象或大而全模块
- 各层强耦合
- 没有证据支撑的过早优化
- 隐藏副作用或未记录依赖
- 必须一次性全部上线才成立的架构
