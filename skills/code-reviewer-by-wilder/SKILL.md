# Wilder 代码审查

代码审查优先关注真实风险，而不是样式噪音。

## 审查顺序

1. 获取变更 diff；如果没有 diff，则检查最近提交。
2. 理解功能或修复的范围。
3. 阅读上下文代码，而不是只看改动片段。
4. 按严重级别审查问题。
5. 只报告高可信、可执行的问题。

## 严重级别

### 严重

- 硬编码密钥或凭据
- 权限绕过或缺少访问控制
- SQL 注入、XSS、路径穿越或 CSRF 暴露
- 日志或响应泄露敏感数据

### 高

- 信任边界缺少输入校验
- 错误路径未处理
- 新行为没有测试覆盖
- 公共接口或契约回归
- 高概率引发缺陷的结构性问题

### 中

- 可能影响用户体验的性能问题
- N+1 查询或无界读取
- 闭包过期或依赖项错误
- 外部调用缺少超时、重试或取消机制

### 低

- 命名不清晰
- 魔法数字
- 公共接口缺少文档
- 没有负责人或上下文的 TODO

## 审查规则

- 优先关注 bug、回归和安全风险，而不是样式偏好。
- 未改动代码通常不审，除非它直接影响本次改动风险。
- 合并重复问题，不要重复刷屏。
- 遵循项目中已经稳定成立的约定。
- 行为发生变化时要指出测试缺失。

## 输出格式

```markdown
[高] 创建接口缺少请求校验
文件: path/to/file.ts:42
问题: 请求体被直接使用，非法输入可能导致运行时失败或状态不一致。
修复: 在进入业务逻辑前使用模式校验，并在失败时返回 4xx 响应。
```

## 总结模板

```markdown
## 审查总结

| 严重级别 | 数量 | 状态 |
|----------|------|------|
| 严重     | 0    | 通过 |
| 高       | 1    | 警告 |
| 中       | 2    | 提示 |
| 低       | 1    | 备注 |

结论: 存在高优先级问题，建议修复后再合并。
```

## 常见检查项

- 信任边界已做输入校验
- 错误处理保留诊断信息但不泄露内部细节
- Hooks、异步流程和状态更新是安全的
- 高成本操作已限界或缓存
- 测试至少覆盖一个成功路径和一个失败路径
