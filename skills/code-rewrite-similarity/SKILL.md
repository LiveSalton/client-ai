# 代码重写与相似度检查

## 目标

指导 AI 按功能推进项目重写：把旧代码作为需求来源，而不是改写模板；基于需求重新设计新实现；使用相似度工具做验收；对高相似片段进行结构性重构；最终生成可追溯的重写记录。

## 核心原则

- 把旧代码用于**需求抽取**，不要把旧代码当作可直接改写的模板。
- 不复制旧代码、注释、日志、测试数据、错误文案、文件结构或函数拆分方式。
- 优先重新设计模块边界、状态模型、数据流、适配层和测试组织方式。
- 相似度工具是质量门禁，不替代法务、版权或工程 Review。
- 不通过简单改变量名、改格式、拆行、移动代码顺序来“降相似度”。

## 工作流

### 1. 确定单个功能范围

先让用户提供或自行推断：

- 旧功能路径或旧功能代码片段
- 新功能目标路径或目标模块
- 使用语言和框架
- 相似度目标；默认建议：`jscpd` 无大段重复，结构相似度低于用户目标，常见目标为 50% 以下

如果用户直接提供代码，就基于当前可见代码继续处理。如果用户提供仓库或文件夹路径，让用户在本地运行本 Skill 附带的脚本，或让用户上传检测报告。

### 2. 从旧代码抽取需求，不生成新代码

只输出需求，不输出实现代码。重点提取：

- 功能目标
- 用户流程
- 输入输出
- 状态流转
- 接口、服务、存储依赖
- UI 行为
- 异常场景
- 测试点

使用 `references/templates.md` 中的“需求说明模板”。

必须明确说明：这些内容是行为需求，不应保留旧代码的命名、结构或实现细节。

### 3. 基于需求重新设计独立实现

在写代码前，先输出新的技术方案。方案应包含：

- 新目录结构
- 新模块职责
- 新命名体系
- 新状态模型
- 新数据流
- 新错误处理方式
- 新测试组织方式

不要沿用旧项目的一一对应文件结构、类名、函数拆分或注释。使用 `references/templates.md` 中的“独立设计模板”。

### 4. 从零生成新代码

根据新的设计方案实现代码。实现时必须：

- 不复用旧代码命名、注释、日志文案、错误文案、mock 数据或测试 case 名称
- 不保持旧函数拆分方式
- 让新代码有独立的模块边界和数据流
- 保持需求层面的功能行为一致
- 补充必要测试

如果用户要求直接修改代码，仍然先确认已完成需求抽取和独立设计；如果上下文中没有这些内容，先补齐再写代码。

### 5. 运行相似度检测

优先使用两类检查：

1. 文本/token 重复：`jscpd`
2. 结构相似：`JPlag`、`MOSS` 或等价工具

本 Skill 提供脚本：

```bash
scripts/check-jscpd.sh <old-path> <new-path> <report-dir> [min-lines] [min-tokens]
```

```bash
scripts/check-jplag.sh <jplag-jar> <language> <report-dir> <old-path> <new-path>
```

如果 AI 无法直接访问本地仓库，就让用户运行脚本并粘贴关键报告、JSON 或截图摘要。

### 6. 分析相似片段并进行结构性重构

拿到报告后，按以下顺序处理：

1. 找出高相似文件和片段
2. 判断相似原因：文本复制、结构相似、通用框架代码、接口约束、业务规则相同
3. 对可重构片段提出结构性修改方案
4. 修改新代码并重新检测

重构必须优先使用这些方式：

- 改变模块边界
- 改变函数拆分
- 改变状态模型
- 改变数据转换流程
- 改变错误处理方式
- 改变组件组织方式
- 改变测试组织方式

不要只做这些无效动作：

- 改变量名
- 改空格、换行、格式化
- 拆行或合并行
- 调换无意义顺序
- 机械替换注释或日志

### 7. 验收标准

不要只看一个 50% 指标。建议使用多维门禁：

| 维度 | 建议标准 |
|---|---|
| `jscpd` 重复率 | 无大段复制，最好低于 5% |
| 连续重复片段 | 单处不超过 20 行连续相似 |
| 结构相似度 | 低于用户目标，常见为 50% 以下 |
| 文件结构 | 不与旧项目一一对应 |
| 命名体系 | 不沿用旧类名、函数名、变量名体系 |
| 文案残留 | 无旧注释、日志、错误文案、mock 数据残留 |
| 人工 Review | 无明显复制旧实现 |

如果因为接口契约、UI 规范或通用框架代码导致无法继续降低相似度，要明确说明原因，并建议从需求、交互、状态机或接口适配层重新设计。

### 8. 输出重写记录

每个功能完成后，生成一份重写记录。使用 `references/templates.md` 中的“重写记录模板”。记录应包含：

- 旧功能参考范围
- 新实现范围
- 独立设计说明
- 相似度检测结果
- 高相似片段处理记录
- 人工 Review 清单
- 剩余风险或不可避免相似点

## 常用提示词

### 需求抽取

```text
请阅读以下旧代码，只提取功能需求和行为说明，不要输出重写代码。
不要复用旧代码结构、命名、注释、函数拆分或实现细节。
请输出：功能目标、用户流程、输入输出、状态流转、依赖接口、异常场景、测试点。
```

### 独立设计

```text
请基于上述需求重新设计实现方案。
要求：不沿用旧目录结构、函数拆分、命名和注释；重新设计模块边界、状态模型和数据流；先输出新文件结构和核心流程，不输出具体代码。
```

### 结构性重构

```text
下面是新旧代码相似度报告和相似片段。
请不要做简单改名、换格式或拆行。
请通过改变模块边界、函数拆分、状态模型、数据转换流程、错误处理和测试组织方式来降低相似度，同时保持功能行为一致。
```
